[WEEK 시스템 메인 메뉴얼]
버전 기준
- 현재 리팩토링 완료 기준
- 주간 루프는 PreTurn -> DayFlow -> NightFlow -> Ending/Next Week 구조
- UI는 WeekFlowViewBase 기반 연결 구조

==================================================
0. 이 프로젝트를 한 줄로 설명하면
==================================================

이 프로젝트는
- ScriptableObject로 주차/정보/이벤트/대화를 정의하고
- RuntimeChildState로 아이의 현재 상태를 저장하고
- WeekRunner가 정보 조절 결과를 처리하고
- WeekNarrativeResolver가 낮 이벤트/밤 대화를 고르고
- WeekFlowController가 전체 흐름을 연결하는
소규모 콘텐츠 중심 Unity 프로젝트다.

핵심 원칙
1. 데이터는 SO에 둔다.
2. 상태는 RuntimeData에 둔다.
3. 흐름은 Flow가 담당한다.
4. 계산은 Systems가 담당한다.
5. UI는 View가 담당한다.
6. 상태 변경은 반드시 SO_CardInteractionDefinition.Apply(...) 또는 GameplayInteractionExecutor.ApplyAll(...)만 사용한다.

==================================================
1. 현재 주간 루프
==================================================

현재 한 주의 진행 순서
1. PreTurn
   - 정보 카드 선택
   - 이번 주 정보 조절 결과 확정
2. DayFlow
   - 정보 조절 결과와 연결된 낮 루틴 이벤트 실행
   - 조건을 만족한 스토리 이벤트 실행
3. NightFlow
   - 조건을 만족한 밤 대화 실행
   - 대화는 1스텝으로 끝날 수도 있고 다음 스텝으로 이어질 수도 있음
4. Ending / Next Week
   - 마지막 주면 엔딩 판정
   - 아니면 다음 주로 이동

중요
- 낮 이벤트, 스토리 이벤트, 밤 대화는 모두 "Interactive Event" 구조로 통일됨
- 기본은 1스텝 이벤트
- 필요하면 여러 스텝으로 이어지는 이벤트 가능

==================================================
2. 가장 중요한 규칙
==================================================

절대 규칙
- RuntimeChildState를 직접 변경하는 코드는 Interaction만 가진다
- childState.AddStat 직접 호출 금지
- childState.SetFlag 직접 호출 금지
- childState.RemoveFlag 직접 호출 금지
- 이벤트/대화/룰/컨트롤러/프레젠터/뷰에서 직접 상태 변경 금지

허용되는 상태 변경 경로
- SO_CardInteractionDefinition.Apply(...)
- GameplayInteractionExecutor.ApplyAll(...)
- RuntimeWeekContext.ApplyInteractions(...)
- SO_WeekRuleDefinition의 ApplyInteractions(...) helper

의미
- 어떤 시스템이든 최종 상태 변경은 반드시 Interaction 배열을 통해 실행한다
- 이벤트가 길어져도, 대화가 이어져도, 상태 변경 규칙은 절대 바뀌지 않는다

5. conditions 설정
6. SO_WeekDayFlowDefinition.RoutineEvents에 연결

의도
- 유저가 조절한 정보와 직접 연관된 1차 피드백용 사건을 만든다

--------------------------------------------------

[스토리 이벤트 추가]
1. SO_StoryEventDefinition asset 생성
2. title / priority / firstStep 설정
3. conditions 설정
4. SO_WeekDayFlowDefinition.StoryEvents에 연결

의도
- 플래그/스탯/정보 조절 결과에 따라 열리는 스토리 전개 이벤트를 만든다

--------------------------------------------------

[밤 대화 추가]
1. SO_PrivateDialogueDefinition asset 생성
2. title / priority / firstStep 설정
3. conditions 설정
4. SO_WeekNightFlowDefinition.Dialogues에 연결

의도
- 밤 대화도 일반 이벤트처럼 관리한다
- 1회 대화도 가능
- 이어지는 대화도 가능

--------------------------------------------------

[이벤트 스텝 추가]
1. SO_InteractiveEventStepDefinition asset 생성
2. bodyText 입력

4. 필요하면 useCustomVisualState 체크

규칙

- choices가 있으면 유저 선택 대기
- 선택지에 nextStep이 있으면 분기 대화 가능

중요

- responseLine은 보여줄 텍스트
- 실제 상태 변화는 interactions

==================================================
10. 조건 작성 규칙
==================================================

현재 조건은 SO_InteractiveEventDefinition.Conditions에 둔다

지원 조건
- RequiredFlags
- BlockedFlags
- StatRequirements
- InformationRequirements

InformationRequirements 의미
- 특정 정보 카테고리가 몇 번 이상 선택되었는가
- 필요하면 semantic(Direct/Modified/Blocked)까지 필터링 가능

예시
- 편지 카테고리에서 Blocked가 1회 이상
- 외부정보 카테고리에서 Direct가 2회 이상
- 방문자 카테고리에서 어떤 결과든 1회 이상

==================================================
11. 지금 꼭 알아야 하는 설계 포인트
==================================================

1. FixedEvent / PrivateDialogue는 더 이상 WeekDefinition 안의 인라인 데이터가 아니다
- 이제 전부 SO로 분리된 이벤트 자산 구조다

2. 모든 이벤트는 1회성으로 가정하지 않는다
- 기본은 1스텝
- 하지만 언제든 여러 스텝으로 이어질 수 있게 설계되어 있다

3. 낮 이벤트 / 스토리 이벤트 / 밤 대화는 모두 같은 인터랙티브 이벤트 구조를 쓴다
- 타입만 다르고 런타임 처리 방식은 거의 같다

4. 상태 변경은 전부 Apply로만 처리한다
- 이벤트가 길어져도 이 원칙은 절대 깨지면 안 된다

14. 빠른 체크리스트
==================================================

이벤트가 안 뜬다
- firstStep이 연결됐는가
- conditions를 너무 빡세게 걸지 않았는가
- NightFlow / DayFlow에 asset이 연결됐는가

이벤트가 1번만 뜨고 안 이어진다
- 선택지 nextStep 또는 step.nextStep이 연결됐는가

선택 후 효과가 안 먹는다
- choice.interactions가 연결됐는가
- step.onEnterInteractions와 choice.interactions를 헷갈리지 않았는가

낮 루틴 이벤트가 정보 조절 결과를 반영하지 않는다
- relatedInformationTypes 설정 여부
- preferredSemantics 설정 여부
- 이번 주 카드의 CardType이 올바른가

밤 대화가 안 나온다
- NightFlow.Dialogues 연결 여부
- priority와 conditions 확인

==================================================
15. 마지막 요약
==================================================

현재 구조는
- WeekDefinition이 주차 루트
- PreTurn이 정보 조절
- DayFlow가 낮 이벤트
- NightFlow가 밤 대화
- InteractiveEvent가 모든 이벤트의 공통 구조
- Apply가 모든 상태 변화의 유일한 입구
라는 원칙으로 정리되어 있다.

이 구조를 유지하면
- 정보 카테고리 확장
- 스토리 이벤트 확장
- 여러 스텝 대화 확장
- 미연시성 방문자 확장
을 무리 없이 계속 얹을 수 있다.
